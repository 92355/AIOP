## 현재까지 진행상황

### 라우트 분리

- `src/app/page.tsx` 중심의 단일 뷰 스위칭 구조를 App Router 라우트 구조로 분리했다.
- 현재 라우트:
  - `/`
  - `/wants`
  - `/calculator`
  - `/regret`
  - `/subscriptions`
  - `/insights`
  - `/notes`
  - `/todos`
- `AppShell`은 `src/app/layout.tsx`에서 공통 shell로 감싸고 있다.

### 내비게이션

- Sidebar, BottomTabBar, SummaryCards는 `Link` 기반 이동으로 바꿨다.
- Header는 현재 pathname 기준으로 제목과 설정 버튼 노출 여부를 계산한다.
- 하위 라우트가 생겨도 active 처리가 유지되도록 route helper를 분리했다.

### QuickAdd / 모달

- QuickAdd 모달은 `AppShell`에서 공통으로 띄운다.
- 저장 후 해당 라우트로 이동하도록 정리했다.
- 업데이트 공지 모달도 추가했다.

### localStorage

- `useLocalStorage`는 같은 탭 이벤트와 다른 탭 storage 이벤트를 함께 듣는다.
- localStorage key 삭제와 `clear()`에 대한 fallback을 넣었다.

### 금액 입력

- 금액 입력 UI를 공통 컴포넌트로 묶었다.
- 적용 위치:
  - 구매 목표 추가
  - 구독 추가
  - 후회 기록 추가
  - 자산 구매 계산기

## `원 / 천 / 만 / 억` 구현 상태

### 적용 방식

- `MoneyInputField`에서 KRW 입력일 때 단위 버튼을 보여준다.
- 버튼을 누르면 현재 입력된 숫자를 선택 단위 기준으로 다시 표시한다.
- 입력값은 내부적으로 원 단위 숫자로 유지한다.
- 입력창 아래에는 원화 미리보기를 보여준다.

### 잘 안되는 지점

#### 1. 단위 변경이 “자동 예측”보다 “수동 선택”에 더 가까움

- 버튼을 누르기 전까지는 자동으로 적절한 단위가 보이지만, 한 번 수동으로 선택하면 그 상태가 유지된다.
- 사용자는 `10000000` 같은 큰 값을 넣었을 때 바로 `1,000` + `만`처럼 명확하게 느끼지 못할 수 있다.

#### 2. 숫자 입력과 단위 버튼의 인지 비용이 남아 있음

- 현재는 입력칸 숫자와 단위 버튼이 분리되어 있어, 사용자가 “지금 이 숫자가 원인지 만인지”를 한 번 더 해석해야 한다.
- 특히 `천 / 만 / 억`을 바꾸면 입력칸 숫자 자체도 바뀌기 때문에, 처음 쓰는 사람은 헷갈릴 수 있다.

#### 3. `USD`와 `KRW`의 인터랙션이 다름

- KRW만 단위 버튼이 보인다.
- 통화를 바꾸면 금액 입력 방식도 같이 바뀌지만, 현재 컴포넌트는 KRW 중심으로 설계되어 있다.

#### 4. 화면마다 숫자 필드 스타일이 아직 완전히 통일되지 않음

- 금액 필드는 공통화됐지만, 목표 기간 / 수익률 / 수량은 아직 일반 숫자 입력이다.
- 그래서 같은 폼 안에서도 어떤 칸은 단위 토글이 있고, 어떤 칸은 없다.

#### 5. 선택된 단위의 상태 복원이 완전하지 않음

- 입력값이 바뀌어도 수동 단위를 계속 유지하는 방식이라, 외부에서 값이 바뀌는 경우 표시 단위가 기대와 다를 수 있다.
- 예를 들어 값이 큰데도 이전에 눌렀던 `원` 상태가 남아 있으면 표시가 덜 읽기 쉬워질 수 있다.

---

## 현재 판단

- “금액을 숫자만 받는 입력”보다는 나아졌지만, 아직은 시각적으로 완성도가 높지 않다.
- 특히 `천 / 만 / 억`은 단순 토글이 아니라 “지금 숫자를 그 단위로 어떻게 읽을지”를 즉시 이해시켜야 하는데, 그 UX가 아직 약하다.
- 다음 개선은 단위 버튼을 입력 안쪽에 더 밀착시키고, 선택 시 입력값/표시값/미리보기를 더 명확히 분리하는 쪽이 적절하다.
